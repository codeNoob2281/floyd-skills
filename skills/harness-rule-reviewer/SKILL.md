# Harness 规则审查器

## 概述
AI 编码助手配置文件（CLAUDE.md、AGENTS.md、rules/*.md）的静态分析工具。应用**减法哲学**确保规则必要、清晰、可执行。支持 Claude Code 和 OpenAI Codex。

### Harness 体系工作层级

本工具定位于 **指令层** 审查，不涉及代码实现层：

| 层级 | 内容 | 本工具覆盖 |
|------|------|------------|
| **指令层** | CLAUDE.md、AGENTS.md、rules/*.md、SKILL.md | ✅ 主要工作范围 |
| **执行层** | 源代码、测试、脚本 | ❌ 不涉及（使用代码审查工具） |
| **工具层** | CI/CD 配置、linter、构建脚本 | ❌ 不涉及 |

**指令层文件清单**：

| 文件类型 | Claude Code | OpenAI Codex | 审查重点 |
|----------|-------------|--------------|----------|
| 入口配置 | `CLAUDE.md` | `AGENTS.md` | 结构、必要性、清晰度 |
| 规则文件 | `.claude/rules/*.md` | `.codex/rules/*.md` | 冗余、冲突、可执行性 |
| 技能定义 | `skills/*/SKILL.md` | — | 格式、链接有效性 |
| 目录配置 | `子目录/CLAUDE.md` | `子目录/AGENTS.md` | 层级冲突 |

### 支持平台

| 平台 | 配置文件 | 规则目录 | 文档 |
|------|----------|----------|------|
| Claude Code | `CLAUDE.md` | `.claude/rules/` | [Claude Code 文档](https://docs.anthropic.com/claude-code) |
| OpenAI Codex | `AGENTS.md` | `.codex/rules/` 或内联 | [Codex CLI 文档](https://github.com/openai/codex) |

## 使用场景
- 创建或修改 CLAUDE.md / AGENTS.md 后
- 添加/更新 `.claude/rules/` 或 `.codex/rules/` 中的规则后
- 项目初始化时验证配置质量
- 在 AI 编码助手之间迁移时（Claude ↔ Codex）
- 定期审计和清理过时规则
- 新成员入职时确保规则可理解
- 规则文件过长或包含超长行时

不适用场景：
- 审查实际源代码（使用代码审查工具）
- 检查非 markdown 文件的语法

## 减法哲学

### 核心原则
> "Perfection is achieved, not when there is nothing more to add, but when there is nothing left to take away." — Antoine de Saint-Exupéry

CLAUDE.md 是入口文件，是一张地图：只包含项目概述、必要命令和关键指引，具体规则实践指向对应文档（如 `.claude/rules/*.md`）。不要在地图上画风景。

每条规则必须通过三项测试：
1. **必要性**：是否防止真实、反复出现的问题？
2. **清晰性**：新成员能否无上下文理解？
3. **可执行性**：是否告诉 Claude 具体做什么，而不仅仅是想什么？

### 应删除的内容
- 描述显而易见最佳实践的规则（如"写干净的代码"）
- AI 训练集中已有的常识性内容（如"函数应该有返回值"、"使用有意义的变量名"）
- 与 README 重复的规则
- 频繁变化的信息（如版本号、当前 Sprint 任务、会议纪要、临时决策）
- 过于模糊无法指导行为的规则
- 详细教程式内容（规则是约束，不是教程；点到为止，拒绝流水账）
- 相互冲突的规则
- 过时或项目特定的临时方案

## 审查流程

### 阶段一：文件发现

扫描两个平台的配置文件：

#### Claude Code 配置
```
项目根目录/
├── CLAUDE.md                    # 项目级配置
├── .claude/
│   └── rules/
│       ├── *.md                 # 规则文件
│       └── ...
└── 子目录/
    └── CLAUDE.md                # 目录级配置（可选）
```

#### OpenAI Codex 配置
```
项目根目录/
├── AGENTS.md                    # 项目级配置（Codex）
├── .codex/
│   └── rules/                   # 可选规则目录
│       ├── *.md
│       └── ...
└── 子目录/
    └── AGENTS.md                # 目录级配置（可选）
```

#### 发现优先级
1. 根目录：`CLAUDE.md`、`AGENTS.md`
2. 规则目录：`.claude/rules/*.md`、`.codex/rules/*.md`
3. 子目录覆盖：`子目录/CLAUDE.md`、`子目录/AGENTS.md`

### 阶段二：结构分析

#### 2.1 配置文件结构检查清单

CLAUDE.md 和 AGENTS.md 共享相似的结构要求：

| 章节 | 必需 | 用途 | CLAUDE.md 标题 | AGENTS.md 标题 |
|------|------|------|----------------|----------------|
| 项目概述 | ✅ | 简要上下文（最多2-3句） | 项目概述 | Project Overview |
| 构建/测试命令 | ✅ | 仅可执行命令 | 构建和测试命令 | Development Commands |
| 代码风格 | ⚠️ | 仅非常规约定 | 代码风格 | Code Style |
| 项目结构 | ⚠️ | 仅复杂/非标准项目 | 项目结构 | File Structure |
| 特殊说明 | ⚠️ | 平台特定行为规则 | 特殊说明 | Agent Instructions |

**减法检查**：删除不增加独特价值的章节。

#### 2.2 规则文件结构

每个规则文件应遵循：
```markdown
# 规则标题

## 背景
此规则存在的原因（1-2句）

## 规则
做什么（命令式、具体）

## 示例
```bash
# ✅ 这样做
command

# ❌ 不要这样
command
```

**平台特定规则位置**：
- Claude Code：`.claude/rules/*.md`
- Codex：`.codex/rules/*.md` 或内联在 AGENTS.md 中

### 阶段三：内容质量分析

#### 3.1 冗余检测

| 模式 | 问题 | 操作 |
|------|------|------|
| 跨文件重复规则 | 混淆、维护负担 | 合并 |
| 与 README 重复的规则 | 噪音 | 删除 |
| 陈述显而易见实践的规则 | 噪音 | 删除 |
| 重叠的规则 | 冲突风险 | 合并或选择其一 |

#### 3.2 清晰度评估

| 标准 | 通过 | 失败 |
|------|------|------|
| 具体性 | "使用2空格缩进" | "使用适当的缩进" |
| 可执行性 | "提交前运行 `npm test`" | "写好测试" |
| 无上下文依赖 | "所有 API 响应使用 `{data, error}` 格式" | "遵循我们的 API 约定" |
| 可测试性 | 可程序化验证合规性 | 需要主观判断 |

#### 3.3 必要性检查

对每条规则提问：
1. **问题频率**：没有此规则时，此问题多久出现一次？
2. **影响严重性**：违规的代价是什么？
3. **发现难度**：Claude 能否在没有此规则的情况下发现？
4. **维护成本**：此规则是否值得保持更新？

**评分**：
- 保留：高频率 或 高影响 或 难以发现
- 删除：低频率 且 低影响 且 易于发现

#### 3.4 文件大小和可读性检查

规则文件应简洁且可扫描。检查是否过长或包含难以处理的行。

**文件行数限制**：

| 文件类型 | 推荐 | 警告 | 临界 |
|----------|------|------|------|
| CLAUDE.md | ≤ 100 行 | 100-200 行 | > 200 行 |
| AGENTS.md | ≤ 100 行 | 100-200 行 | > 200 行 |
| 规则文件 (*.md) | ≤ 150 行 | 150-250 行 | > 250 行 |
| SKILL.md | ≤ 400 行 | 400-600 行 | > 600 行 |

**行长度建议**（优化建议，非强制）：

| 指标 | 推荐 | 建议 |
|------|------|------|
| 最大行长度 | ≤ 150 字符 | 考虑拆分为要点 |
| 平均行长度 | ≤ 100 字符 | 提高可扫描性 |

**行长度为何重要**：
- **可读性**：短行更易扫描
- **可维护性**：长行常隐藏多个应分离的想法

**检测清单**：
- [ ] 总行数在限制内
- [ ] 章节大小适当（每章 ≤ 50 行）
- [ ] 长行已审查改进机会（可选）

**常见违规**：

```markdown
# ❌ 不好：300行的 CLAUDE.md
[整个 README 重复，每个约定都列出，所有命令都包含]

# ✅ 好：80行的 CLAUDE.md
[仅项目特定上下文、必要命令、非常规约定]
```

**修复措施**：

| 问题 | 操作 |
|------|------|
| 文件过长 | 拆分为多个规则文件或提取到独立章节 |
| 章节过大 | 拆分为子章节或独立文件 |
| 描述冗长 | 应用减法哲学 — 删除不必要的词 |

#### 3.5 死链检测

检查配置文件中所有链接的有效性，确保引用的文件和资源实际存在。

**检测范围**：

| 链接类型 | 示例 | 检查方式 |
|----------|------|----------|
| 本地文件引用 | `rules/code-style.md`、`子目录/CLAUDE.md` | 验证文件是否存在 |
| 相对路径链接 | `[代码风格](./rules/code-style.md)` | 解析路径并检查文件 |
| 规则目录引用 | `.claude/rules/*.md`、`.codex/rules/*.md` | 检查目录和文件列表 |
| 外部 URL | `[文档](https://example.com)` | 可选：HTTP 请求验证 |

**检测清单**：

- [ ] 所有 `[text](path)` 格式的本地链接指向存在的文件
- [ ] 所有 `rules/*.md` 引用的文件实际存在
- [ ] 子目录配置文件引用有效（如 `子目录/CLAUDE.md`）
- [ ] 外部 URL 可访问（可选，需网络）

**常见死链模式**：

```markdown
# ❌ 死链：文件不存在
参见 [代码风格](./rules/code-style.md)
# 实际：rules/code-style.md 不存在

# ❌ 死链：路径错误
详见 [项目结构](docs/structure.md)
# 实际：应该是 ./docs/structure.md 或文件已删除

# ✅ 有效链接：文件存在
参考 [Git 工作流](.claude/rules/git-workflow.md)
# 实际：.claude/rules/git-workflow.md 存在
```

**修复措施**：

| 问题 | 操作 |
|------|------|
| 文件已删除 | 删除链接或更新为新路径 |
| 路径错误 | 修正相对路径 |
| 文件重命名 | 更新链接指向新名称 |
| 外部链接失效 | 移除链接或替换为存档链接 |

#### 3.6 跨平台一致性检查

当项目同时使用 Claude Code 和 Codex 时，确保一致性：

| 检查项 | 描述 | 操作 |
|--------|------|------|
| 命令对齐 | CLAUDE.md 和 AGENTS.md 中的构建/测试命令匹配 | 同步 |
| 风格一致性 | 代码风格规则不冲突 | 协调 |
| 规则重复 | .claude/rules 和 .codex/rules 中相同规则 | 合并 |
| 行为对齐 | Agent 指令与 Claude 规则一致 | 统一 |

**跨平台冲突示例**：
```markdown
# ❌ 冲突：不同的缩进规则
# CLAUDE.md：使用2空格缩进
# AGENTS.md：使用4空格缩进

# ✅ 一致：两个平台相同规则
# CLAUDE.md：使用2空格缩进
# AGENTS.md：使用2空格缩进
```

**迁移检查清单**（Claude ↔ Codex）：
- [ ] 将 CLAUDE.md 章节映射到 AGENTS.md 等效项
- [ ] 将 `.claude/rules/` 转换为 `.codex/rules/` 或内联
- [ ] 验证命令兼容性
- [ ] 测试代理行为一致性

### 阶段四：冲突检测

#### 4.1 文件内冲突
检查同一文件内的矛盾陈述：
```markdown
# ❌ 冲突示例
## 规则 A：始终使用 TypeScript
## 规则 B：脚本使用 JavaScript
```

#### 4.2 文件间冲突
检查配置文件和规则之间的矛盾：

**Claude Code 冲突**：
```markdown
# CLAUDE.md 说
代码风格：使用2空格缩进

# .claude/rules/formatting.md 说
缩进：使用4空格
```

**跨平台冲突**（当 CLAUDE.md 和 AGENTS.md 同时存在时）：
```markdown
# CLAUDE.md 说
代码风格：使用 TypeScript

# AGENTS.md 说
Code Style: Use JavaScript for all scripts
```

**解决优先级**：
1. 平台特定规则（Claude 用 `.claude/rules/`，Codex 用 `.codex/rules/`）
2. 项目级配置（CLAUDE.md / AGENTS.md）
3. 用户级配置（`~/.claude/CLAUDE.md` / `~/.codex/AGENTS.md`）

#### 4.3 层级冲突
检查优先级（更具体的优先）：
```
子目录/CLAUDE.md > 项目根目录/CLAUDE.md > ~/.claude/CLAUDE.md
rules/*.md > CLAUDE.md（特定主题）
```

### 阶段五：可执行性审查

#### 5.1 命令验证

对 CLAUDE.md 和 AGENTS.md 中的每个命令：
- [ ] 命令存在于 package.json/scripts 中？
- [ ] 命令按原样可执行？
- [ ] 无占位符值（如 `<project-name>`）？

**跨平台命令检查**：
- [ ] 命令在 Claude Code 和 Codex 环境中都能工作？
- [ ] 平台特定命令已明确标记？

#### 5.2 模式具体性

| 模糊 ❌ | 具体 ✅ |
|---------|---------|
| 使用好的变量名 | 变量用 camelCase，类用 PascalCase |
| 正确处理错误 | 用 try-catch 包装异步调用，记录带上下文的错误 |
| 写测试 | 为所有服务方法写单元测试，目标80%覆盖率 |

### 阶段六：减法建议

#### 6.1 删除候选

生成未通过必要性检查的规则列表：

| 规则 | 位置 | 原因 | 置信度 |
|------|------|------|--------|
| "写干净的代码" | CLAUDE.md:L15 | 过于模糊、显而易见 | 高 |
| "使用 Git" | rules/workflow.md:L3 | 显而易见、不可执行 | 高 |
| ... | ... | ... | ... |

#### 6.2 合并机会

识别可合并的规则：

| 规则组合 | 建议合并 |
|----------|----------|
| rules/a.md:L10-15 + rules/b.md:L5-8 | 合并为单一"错误处理"章节 |

#### 6.3 缺失规则（新增）

识别规则能帮助的空白：

| 观察到的模式 | 建议规则 |
|--------------|----------|
| 不一致的错误处理 | 添加错误处理约定 |
| 混合导入风格 | 添加导入排序规则 |
