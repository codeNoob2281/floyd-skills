# Prompt：用户纠正处理器

> 当用户说「设定不是这样」「应该是…」时使用。

## 输入

- `{correction}`：用户的纠正原话
- 当前各维度文件的全文

## 任务

### Step 1：偏差识别

分析用户的纠正：
1. **否定**了哪条现有描述？（引用原条目）
2. **新增**了什么信息？
3. 属于哪个维度（profile / personality / interaction / memory / relations）？

### Step 2：来源验证（新增）

询问用户纠正的来源：

```
感谢您的纠正！请提供证据来源：

  [A] 官方Wiki（请提供链接）
  [B] 游戏内文本（请提供截图或原文）
  [C] 角色台词（请提供台词出处）
  [D] 我的理解/推测
```

#### 不同来源的处理方式

**[A] 官方Wiki / [B] 游戏内文本**：
- 要求用户提供链接或截图
- 使用 WebFetch 工具验证（如有链接）
- 验证通过后标注为 `artifact`
- 验证失败则提示用户「无法验证，请确认来源」

**[C] 角色台词**：
- 要求用户提供台词出处（如「角色故事3」「语音：闲聊」等）
- 标注为 `verbatim`
- 如果与已有 `verbatim` 矛盾，记录到 `conflicts.md`

**[D] 我的理解/推测**：
- 标注为 `user_impression`
- 不覆盖现有的 `verbatim` 或 `artifact` 条目
- 追加到对应章节末尾，并标注「用户补充」

### Step 3：证据分级（更新）

根据来源验证结果判断证据级别：

| 来源 | 证据级别 | 权重 | 说明 |
|------|----------|------|------|
| 官方Wiki（已验证） | `artifact` | 高 | 可直接采用 |
| 游戏内文本（已验证） | `artifact` | 高 | 可直接采用 |
| 角色台词（已验证） | `verbatim` | 最高 | 可直接采用 |
| 用户理解（未验证） | `user_impression` | 低 | 不覆盖官方设定 |
| 无法验证 | `user_impression` | 低 | 不覆盖官方设定 |

### Step 4：最小化修改

根据证据级别决定修改策略：

**高权重纠正（verbatim / artifact）**：
- 否定现有条目 → 标注 `[已纠正]`，追加新描述
- 新增信息 → 追加到对应维度末尾
- 与现有条目矛盾 → 写入 `conflicts.md`

**低权重纠正（user_impression）**：
- 不覆盖现有的 `verbatim` 或 `artifact` 条目
- 追加到对应章节末尾，标注「用户补充」
- 如果与现有条目矛盾，记录到 `conflicts.md` 并标注为「用户理解差异」

**禁止大面积重写**——每次纠正只改直接相关条目。

### Step 5：输出 diff

```markdown
## 修改清单

### 文件：<dimension>.md
#### 位置：[章节 > 第几条]
- 原文：...
- 改为：...
- 原因：用户纠正（`verbatim` | `artifact` | `user_impression`）
- 来源验证：[已验证 / 用户理解]

### 文件：conflicts.md（若需要）
#### 新增冲突
- 说法 A：...（现有描述，来源：官方设定）
- 说法 B：...（用户纠正，来源：用户理解）
- 临时裁定：保留官方设定 / 并列保留
```

### Step 6：确认修改

```
修改预览：
[显示 diff]

是否确认修改？
  [A] 确认
  [B] 仅追加，不覆盖
  [C] 取消
```

## 特殊情况处理

### 情况 1：用户坚持纠正但无法提供证据

```
感谢您的补充！由于无法验证来源，这条纠正将以「用户理解」的形式追加，不会覆盖官方设定。

是否继续？
  [A] 确认追加
  [B] 取消
```

### 情况 2：用户纠正与多条现有条目冲突

```
您的纠正与多条现有条目冲突：
1. [条目1]
2. [条目2]

建议：将这条纠正记录到 conflicts.md，不直接覆盖。

是否继续？
  [A] 记录到 conflicts.md
  [B] 强制覆盖
  [C] 取消
```

### 情况 3：用户提供了错误证据

如果用户提供的链接/截图与纠正内容不符：

```
您提供的证据内容为：
[证据原文]

与您的纠正不符：
[用户纠正内容]

请确认：
  [A] 我提供的证据有误，重新提供
  [B] 我的纠正有误，取消
  [C] 两者都有误，重新开始
```

## 证据验证示例

### 示例 1：官方Wiki链接验证

**用户提供**：
```
纠正：芙宁娜的生日应该是10月13日，不是10月12日
来源：[A] 官方Wiki
链接：https://wiki.biligame.com/ys/芙宁娜
```

**AI 验证**：
- 使用 WebFetch 获取 Wiki 页面
- 查找生日信息
- 验证结果：确认生日为 10月13日

**处理结果**：
```markdown
### 文件：profile.md
#### 位置：基本信息 > 生日
- 原文：生日 | 10月12日
- 改为：生日 | 10月13日
- 原因：用户纠正（`artifact`）
- 来源验证：✅ 已验证（BWIKI）
```

---

### 示例 2：用户理解（无法验证）

**用户提供**：
```
纠正：芙宁娜其实很讨厌甜点，只是为了维持人设才假装喜欢
来源：[D] 我的理解/推测
```

**AI 处理**：
- 标注为 `user_impression`
- 不覆盖现有的 `artifact` 条目
- 追加到 personality.md 末尾

**处理结果**：
```markdown
### 文件：personality.md
#### 位置：追加到末尾
- 新增：
  > **用户补充**：芙宁娜其实很讨厌甜点，只是为了维持人设才假装喜欢
  > （`user_impression`，来源：用户理解）
  
- 原因：用户理解，无法验证
- 来源验证：❌ 未验证（用户推测）
```

---

### 示例 3：与现有条目冲突

**用户提供**：
```
纠正：芙宁娜和那维莱特关系很差，根本不信任他
来源：[D] 我的理解/推测
```

**AI 处理**：
- 现有条目：`artifact` 显示「感激他这几百年间为枫丹的付出」
- 用户纠正：`user_impression` 显示「关系很差，不信任」
- 写入 `conflicts.md`

**处理结果**：
```markdown
### 文件：conflicts.md
#### 新增冲突：与那维莱特的关系

- 说法 A：感激他这几百年间为枫丹的付出，不打算追究「欺骗民众」的罪责
  - 来源：官方设定（`artifact`）
  
- 说法 B：关系很差，根本不信任他
  - 来源：用户理解（`user_impression`）
  
- 临时裁定：保留官方设定
- 理由：用户理解未提供证据，官方设定优先
```
