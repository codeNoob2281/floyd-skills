# 增量更新机制

> 本文档定义如何检测游戏版本更新并增量更新角色蒸馏结果。
> 目标：避免完全重新蒸馏，只更新变化的部分。

---

## 一、更新触发条件

### 自动触发

| 触发条件 | 检测方式 | 频率 |
|----------|----------|------|
| 游戏版本更新 | 检测 `source_version` 与当前版本差异 | 用户蒸馏时 |
| Wiki 内容更新 | 对比 Wiki 页面最后更新时间 | 用户请求时 |
| 设定冲突增加 | `conflicts.md` 新增条目超过阈值 | 实时 |

### 手动触发

用户主动请求更新：

```
「帮我更新芙宁娜的设定」
「重新蒸馏钟离」
```

---

## 二、版本检测流程

### Step 1：读取现有版本

从 `manifest.json` 读取：

```json
{
  "source_version": "4.2",
  "last_updated": "2026-03-15T10:00:00Z"
}
```

### Step 2：获取当前版本

从官方渠道获取当前游戏版本：

| 游戏 | 版本来源 | 检测方式 |
|------|----------|----------|
| 原神 | 官方公告 / Wiki 首页 | WebFetch 获取 |
| 星铁 | 官方公告 / Wiki 首页 | WebFetch 获取 |
| 三角洲行动 | 官方公告 / BWIKI | WebFetch 获取 |

### Step 3：对比版本

```python
if current_version != source_version:
    # 检测到版本更新
    prompt = f"""
    检测到版本更新：
    - 当前版本：{current_version}
    - 已蒸馏版本：{source_version}
    
    是否更新角色设定？
    [A] 完整更新（推荐）
    [B] 增量更新（仅更新变化部分）
    [C] 跳过
    """
```

## 三、增量更新流程

### 完整更新（推荐）

重新执行完整蒸馏流程：

1. 重新获取 Wiki 内容
2. 重新提取五个维度
3. 重新生成所有文件
4. 重置 `conflicts.md`
5. 更新 `manifest.json`

**适用场景**：
- 大版本更新（如 4.0 → 5.0）
- 角色设定大幅修改
- 距离上次更新超过 3 个月

---

### 增量更新

只更新变化部分：

#### Step 1：获取更新内容

```
请提供新增或修改的设定内容：
  [A] 自动获取Wiki更新（推荐）
  [B] 手动粘贴新增内容
  [C] 上传更新文件
```

#### Step 2：对比变化

对比新旧内容，识别：

| 变化类型 | 处理方式 |
|----------|----------|
| 新增设定 | 追加到对应维度文件 |
| 修改设定 | 标注为 `[已更新]`，保留旧版本 |
| 删除设定 | 标注为 `[已废弃]`，保留记录 |
| 冲突设定 | 记录到 `conflicts.md` |

#### Step 3：更新维度文件

**新增内容处理**：

```markdown
## [章节名]

### [新增条目]
- [内容] （`artifact`，新增于版本 X.X）

---
> 📅 新增于版本 X.X
```

**修改内容处理**：

```markdown
### [已修改条目]

**旧版本**（< X.X）：
- [旧内容] （`artifact`）

**新版本**（≥ X.X）：
- [新内容] （`artifact`）

---
> 📅 更新于版本 X.X
```

**废弃内容处理**：

```markdown
### [已废弃条目]
- [内容] （`artifact`，已废弃）

~~这条设定已在版本 X.X 中被废弃。~~
```

#### Step 4：更新 conflicts.md

如果新旧设定矛盾：

```markdown
## 冲突：[冲突主题]

### 版本 < X.X
- [旧设定]

### 版本 ≥ X.X
- [新设定]

### 处理方案
- 采用：新版本设定
- 理由：官方已明确修改
```

#### Step 5：更新 manifest.json

```json
{
  "source_version": "4.3",
  "last_updated": "2026-04-05T08:30:00Z",
  "update_history": [
    {
      "version": "4.2",
      "date": "2026-03-15T10:00:00Z",
      "type": "完整蒸馏"
    },
    {
      "version": "4.3",
      "date": "2026-04-05T08:30:00Z",
      "type": "增量更新",
      "changes": [
        "新增角色故事章节",
        "修改了与那维莱特的关系描述"
      ]
    }
  ]
}
```

## 四、更新策略

### 不同版本差异的策略

| 版本差异 | 更新策略 | 说明 |
|----------|----------|------|
| 主版本更新（4.x → 5.x） | 完整更新 | 可能有世界观变化 |
| 次版本更新（4.2 → 4.3） | 增量更新 | 通常只有新内容 |
| 补丁更新（4.2.1 → 4.2.2） | 跳过 | 通常不影响角色设定 |

### 不同角色的更新优先级

| 角色类型 | 更新优先级 | 说明 |
|----------|------------|------|
| 主线角色 | 高 | 剧情推进会影响设定 |
| 活动角色 | 中 | 可能有新剧情 |
| NPC | 低 | 设定相对稳定 |

## 五、更新后验证

### 自动验证

更新后自动执行：

1. **完整性检查**：五个维度文件是否都存在
2. **格式检查**：证据标注是否完整
3. **冲突检查**：是否新增了未记录的冲突

### 扮演测试

更新后建议重新执行扮演测试：

```
更新完成，是否重新测试角色扮演？
  [A] 是（推荐）
  [B] 否
```

## 六、版本回滚

### 何时需要回滚

- 更新后发现设定错误
- 新版本设定不如旧版本完整
- 用户不满意更新结果

### 回滚流程

#### Step 1：查找历史版本

```bash
ls -la characters/furina/versions/
# 4.2_2026-03-15/
# 4.1_2026-02-01/
```

#### Step 2：恢复旧版本

```
确认回滚到版本 4.2？
  [A] 确认
  [B] 取消
```

#### Step 3：记录回滚

在 `manifest.json` 中记录：

```json
{
  "rollback_history": [
    {
      "from_version": "4.3",
      "to_version": "4.2",
      "date": "2026-04-05T09:00:00Z",
      "reason": "新版本设定错误"
    }
  ]
}
```

## 七、版本管理最佳实践

### 命名规范

```
characters/<slug>/
├── current/              # 当前版本
│   ├── SKILL.md
│   ├── profile.md
│   └── ...
└── versions/             # 历史版本
    ├── 4.2_2026-03-15/
    └── 4.1_2026-02-01/
```

### 保留版本数量

- 默认保留最近 3 个版本
- 主版本更新时，保留上一个主版本的最后一个次版本

### 版本标记

在 `SKILL.md` 的 frontmatter 中标记：

```yaml
---
name: furina
version: "4.3"
last_updated: 2026-04-05
---
```

是否更新角色设定？

## 九、注意事项

1. **版本号格式**：统一使用 `主版本.次版本.补丁` 格式（如 4.3.1）
2. **更新频率**：建议每 1-2 个版本更新一次，避免积累过多变化
3. **备份习惯**：重大更新前建议备份现有版本
4. **测试验证**：更新后务必重新执行扮演测试
