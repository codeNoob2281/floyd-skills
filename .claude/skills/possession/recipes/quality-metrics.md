# 质量评分标准

> 本文档定义角色蒸馏结果的质量评估标准与计算方法。
> 目标：量化评估蒸馏质量，为用户提供改进建议。

---

## 一、评分维度

### 1. 完整度（Completeness）

**定义**：五个维度的覆盖完整程度

**计算方式**：
```
完整度 = 非空维度数 / 5

其中「非空维度」定义为：
- 至少有 3 条有效内容
- 每条内容都有证据标注
```

**评分标准**：

| 分数范围 | 评级 | 说明 |
|----------|------|------|
| ≥0.9 | 优秀 | 五维度全覆盖，内容丰富 |
| 0.7-0.89 | 良好 | 大部分维度完整 |
| 0.6-0.69 | 及格 | 基本维度覆盖，但内容较少 |
| <0.6 | 不合格 | 维度缺失严重 |

**权重**：30%

---

### 2. 证据率（Evidence Ratio）

**定义**：高质量证据（verbatim + artifact）占总条目的比例

**计算方式**：
```
证据率 = (verbatim 条目数 + artifact 条目数) / 总条目数
```

**评分标准**：

| 分数范围 | 评级 | 说明 |
|----------|------|------|
| ≥0.8 | 优秀 | 大部分内容有官方设定支撑 |
| 0.7-0.79 | 良好 | 多数内容有证据 |
| 0.5-0.69 | 及格 | 约一半内容有证据 |
| <0.5 | 不合格 | 缺乏官方设定支撑 |

**权重**：40%

---

### 3. 冲突数（Conflict Count）

**定义**：`conflicts.md` 中记录的设定冲突数量

**评分方式**：
```
冲突数得分 = 
  0 冲突 → 1.0
  1-2 冲突 → 0.9
  3-5 冲突 → 0.7
  6-10 冲突 → 0.5
  >10 冲突 → 0.3
```

**说明**：
- 冲突数少不代表质量高（可能是设定本身就一致）
- 冲突数多需要关注，可能影响角色扮演一致性

**权重**：10%

---

### 4. 测试通过率（Test Pass Rate）

**定义**：扮演测试通过场景占总测试场景的比例

**计算方式**：
```
测试通过率 = 通过场景数 / 8

其中「通过场景」定义为：
- 单场景得分 ≥ 0.7
- 无关键 OOC 问题
```

**评分标准**：

| 分数范围 | 评级 | 说明 |
|----------|------|------|
| ≥0.9 | 优秀 | 扮演一致性高 |
| 0.8-0.89 | 良好 | 大部分场景通过 |
| 0.7-0.79 | 及格 | 基本场景通过，有轻微问题 |
| <0.7 | 不合格 | 存在明显 OOC 问题 |

**权重**：20%

---

## 二、综合评分计算

### 计算公式

```
总评分 = 完整度 × 0.3 
       + 证据率 × 0.4 
       + 冲突数得分 × 0.1 
       + 测试通过率 × 0.2
```

### 评级标准

| 总评分 | 评级 | 处理建议 |
|--------|------|----------|
| ≥0.85 | 优秀 ⭐⭐⭐⭐⭐ | 可直接使用，角色设定完整且一致 |
| 0.70-0.84 | 良好 ⭐⭐⭐⭐ | 可使用，建议查看 conflicts.md |
| 0.60-0.69 | 及格 ⭐⭐⭐ | 建议补充设定材料后重新蒸馏 |
| <0.60 | 不合格 ⭐⭐ | 必须补充材料后重新蒸馏 |

## 三、评分示例

### 示例 1：芙宁娜（优秀）

```json
{
  "quality_score": {
    "completeness": 1.0,
    "evidence_ratio": 0.87,
    "conflict_count": 0,
    "conflict_score": 1.0,
    "test_pass_rate": 0.875,
    "overall": 0.925
  },
  "rating": "优秀",
  "summary": "五维度全覆盖，87%内容有官方设定支撑，无设定冲突，扮演测试通过率87.5%"
}
```

**计算过程**：
- 完整度：5/5 = 1.0
- 证据率：(40 verbatim + 43 artifact) / 95 总条目 = 0.87
- 冲突数得分：0 冲突 → 1.0
- 测试通过率：7/8 = 0.875
- 总评分：1.0×0.3 + 0.87×0.4 + 1.0×0.1 + 0.875×0.2 = 0.925

---

### 示例 2：某角色（及格）

```json
{
  "quality_score": {
    "completeness": 0.6,
    "evidence_ratio": 0.45,
    "conflict_count": 3,
    "conflict_score": 0.7,
    "test_pass_rate": 0.625,
    "overall": 0.55
  },
  "rating": "不合格",
  "summary": "缺少2个维度，仅45%内容有证据支撑，存在3个设定冲突，扮演测试通过率62.5%"
}
```

**计算过程**：
- 完整度：3/5 = 0.6
- 证据率：(15 + 12) / 60 = 0.45
- 冲突数得分：3 冲突 → 0.7
- 测试通过率：5/8 = 0.625
- 总评分：0.6×0.3 + 0.45×0.4 + 0.7×0.1 + 0.625×0.2 = 0.55

## 四、自动补充建议

### 触发条件

当评分 < 0.70 时，自动生成补充建议。

### 建议生成规则

| 问题 | 建议内容 |
|------|----------|
| 完整度 < 0.6 | 「缺少 {缺失维度} 维度，请提供相关设定材料」 |
| 证据率 < 0.5 | 「{低证据维度} 缺少官方设定支撑，请补充角色台词或官方文本」 |

## 七、特殊情况处理

### 情况 1：未执行扮演测试

如果用户跳过扮演测试，质量评分为：

```
总评分 = 完整度 × 0.4 + 证据率 × 0.5 + 冲突数得分 × 0.1
```

并在报告中标注：「未执行扮演测试，评分可能偏高」

---

### 情况 2：设定本身较少

对于设定较少的角色（如NPC），降低评分标准：

| 角色类型 | 及格线 | 良好线 |
|----------|--------|--------|
| 主要角色 | 0.60 | 0.70 |
| 次要角色 | 0.50 | 0.60 |
| NPC | 0.40 | 0.50 |

---

### 情况 3：多版本角色

对于存在多个版本的角色（如原神的「旅行者」），在报告中标注：

```
⚠️ 本角色存在多版本设定，已按以下原则处理：
- 采用最新版本设定
- 旧版本设定记录在 conflicts.md
- 版本差异已标注在对应条目
```
